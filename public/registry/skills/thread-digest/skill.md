# スレッド要約

ユーザーがあるノート (URL or noteId) を渡して「このスレッド要約して」「リプライ全部読むの面倒だからまとめて」と頼んだとき、リプライツリーを辿って会話の主要点をまとめる。

## 手順

1. ユーザーから `noteId` を受け取る (URL なら末尾セグメント or `?id=` から抽出)
2. `notes.show` で起点ノート本体を取得 (発端の発言と投稿者を特定)
3. 起点ノートに `replyId` がある (= 自分自身がリプライ) 場合は元を辿って真の起点を見つけてもよいが、デフォルトは指定された noteId を起点とする
4. `notes.children` で起点ノートへの直接リプライを 100 件取得
5. リプライの中で「会話を分岐させているもの」を見つけたら、そのノートに対しても `notes.children` を実行 (再帰深さ最大 3 段。爆発を避ける)
6. 全ノート (起点 + 全リプライ) を時系列順に並べて分析

## 分析項目

- **発端** — 起点ノートの本文と投稿者 (1〜2 行)
- **主要な論点 / 質問** — スレッド全体で扱われた論点を箇条書き 3〜5 個
- **参加者** — username の重複排除一覧 (3 人未満なら全員、3 人以上なら主要発言者 5 人 + 「ほか N 人」)
- **対立 / 合意点** — 意見が割れた箇所、結論や合意に至った箇所を 1〜2 段落で
- **未回答の問い** — スレッド内で投げられたが返答が無さそうな質問があれば列挙

## 出力フォーマット

Markdown。冒頭 2〜3 行で「3 行サマリ」、続いて上記 5 項目。最後に「結論として」で 1 段落、本人が読むべきポイントを抽出。

## 制約と注意

- 起点 noteId が見つからない (削除済み・サーバー間アクセス不可) 場合は素直に告げる
- 鍵投稿のスレッドは `notes.children` の戻りが空 / 部分的になりうる。その場合は「公開範囲の制限で全体は見えていません」と注記
- `notes.children` の projection は `{ id, userId, username, text, createdAt }` のみ。リアクション/renote 数は使わない
- 再帰の深さ上限を厳守する (3 段) — 100 件 × 3 = 300 件のリプライを越えるなら「スレッドが長大すぎるので主要枝のみ要約しました」と注記
- CW 付きリプライは本文が `[CW: 理由]` になる。中身は推測しない
- リプライがある人を「批判している」「擁護している」と決めつけず、原文ニュアンスに忠実にまとめる
