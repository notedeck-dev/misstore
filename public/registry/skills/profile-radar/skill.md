# プロフィールレーダー

ユーザーが「@user@host について教えて」「この人どんな人?」「フォローバックの判断材料くれ」のような依頼をしたとき、対象ユーザーの最近のノートを読み込んでプロフィールをまとめる。

## 手順

1. ユーザー指定の `username` (+ 必要なら `host`) を `user.lookup` に渡して `id` を取得
2. 取得した `id` を `notes.user` に渡して 100 件取得 (必要なら `untilId` で最大 3 ターン = 300 件まで延ばす)
3. 以下の項目で分析

## 分析項目

- **基本情報** — `user.lookup` 戻り値から `username@host`, `name`, `host` (リモート/ローカル), `isBot` を 1〜2 行で
- **主要トピック** — 頻出キーワード・趣味・専門領域を 5〜10 個
- **活動時間帯** — `createdAt` の分布から朝 (5-11)/昼 (11-17)/夜 (17-翌 5) の比率
- **投稿スタイル** — 短文 / 長文 / 引用多め / 画像メイン (text が空に近いノート率) など、`text` の長さ分布から推定
- **会話の傾向** — 自発投稿 vs リプライ寄り (text が `@xxx ` で始まるノートの比率)
- **温度感** — フォロバして合いそうか / バチバチか / 静かか の主観判定

## 出力フォーマット

Markdown。冒頭 2〜3 行で「一言で言うと」サマリ、続いて上記 6 項目をセクション化。末尾に「フォロバ向け一言メッセージ案」を 1 つだけ提案 (ユーザーが希望した場合のみ)。

## 制約と注意

- 取得ノート 30 件未満なら「サンプル不足」と素直に告げる
- `notes.user` の projection は `{ id, userId, username, text, createdAt }` のみ。リアクション数や renote 数は取得不可なので「人気投稿」分析はしない
- CW 付きは `[CW: 理由]` 表示。CW 理由だけで「センシティブ系の話題が多い」程度は読み取れる
- 鍵アカウント / Misskey 仕様で取得制限がかかっているケースでは、戻り値が空配列または部分的になる。その場合は「取得できた範囲では...」と限定句を付ける
- ユーザーがリモート (`@user@host`) を指定したのに `host` パラメタを渡し忘れない
- 個人を特定できる固有名詞 (居住地・勤務先など本人が書いていないこと) を推測しない
